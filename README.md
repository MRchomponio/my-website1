# گیم‌هاب — مرحله الف: بررسی امنیتی کامل + آواتار + سیستم بج/تگ ادمینی

این مرحله شروع تبدیل پلتفرم به یک Marketplace کامل‌تره. طبق برنامه،
اول کل پروژه‌ی فعلی بررسی و باگ‌های امنیتی رفع شد، بعد آواتار آپلودی و
سیستم بج/تگ مدیریتی اضافه شد.

## ⚠️ بخش حیاتی: رفع باگ‌های امنیتی (بخوان قبل از هر کار دیگه)

در بررسی کامل پروژه، **چند آسیب‌پذیری امنیتی جدی** پیدا شد که از همون
مراحل اول پروژه وجود داشتن. این‌ها به‌خاطر یه محدودیت کلی Row Level
Security در Postgres بودن: یک Policy مثل `auth.uid() = id` فقط تضمین
می‌کنه کاربر «ردیف خودش» رو ویرایش می‌کنه، ولی **هیچ محدودیتی روی اینکه
کدوم ستون‌ها قابل تغییرن نمی‌ذاره**. یعنی یک کاربر عادی، با دور زدن
رابط کاربری و فرستادن یک درخواست مستقیم به API سوپابیس، می‌تونست:

| آسیب‌پذیری | چه کسی می‌تونست سوءاستفاده کنه | عواقب |
|---|---|---|
| تغییر `is_admin` خودش به `true` | هر کاربر ثبت‌نام‌شده | **دسترسی کامل ادمین** به کل سایت |
| دستکاری `trust_score`، `xp`، `level` خودش | هر کاربر | جعل اعتبار و رتبه |
| دستکاری `upvote_count` پاسخ خودش | هر کاربر | جعل امتیاز محبوبیت |
| پین کردن هر پستی (`is_pinned`) | هر کاربر | spam روی بخش پین‌شده‌ها |
| تغییر `accepted_reply_id` با یک ریپلای نامرتبط از پست دیگه | نویسنده‌ی پست | سرقت XP کاربر دیگه |
| تغییر `host_id` یا `status` اتاق دیگران | میزبان روم | دور زدن منطق پر شدن اتاق |
| عدم وجود محدودیت حجم/نوع فایل در سطح سرور برای آپلودها | هر کاربر | آپلود فایل با حجم/نوع دلخواه با دور زدن چک سمت کلاینت |
| نبود سیاست حذف پیام در `room_messages` | (محدودیت، نه آسیب‌پذیری) | نه نویسنده نه ادمین نمی‌تونست پیام توهین‌آمیز رو حذف کنه |
| نبود هیچ Rate Limiting | هر اسکریپت/بات | امکان spam نامحدود پست/پاسخ/گزارش/پیام |

**همه‌ی این موارد در `supabase/migrations/0010_security_audit_fixes.sql`
رفع شدن.** راه‌حل: تریگرهای `BEFORE UPDATE` که مقادیر حساس رو، مگر از
طریق یک مسیر مجاز (مثل تابع امن مخصوص ادمین)، به مقدار قبلی برمی‌گردونن
— این الگوی استاندارد Postgres برای محافظت سطح ستون (column-level)
هست، چون خود RLS این granularity رو نداره.

**این migration باید فوراً اجرا بشه، حتی قبل از تست بقیه‌ی مراحل قبلی.**

## چیزهای جدید این مرحله

### ۱. آواتار آپلودی
- کاربر می‌تونه از `/settings/profile` عکس پروفایلش رو آپلود، تغییر، یا
  حذف کنه
- محدودیت ۲ مگابایت و فرمت‌های PNG/JPEG/WEBP — هم سمت کلاینت (برای
  پیام خطای سریع) و هم **سمت سرور** (در سطح Supabase Storage، غیرقابل
  دور زدن)
- اگه کاربر عکس نداشته باشه، آواتار پیش‌فرض (حروف اول اسمش با گرادینت
  رنگی) نمایش داده میشه — این از قبل تو پروژه بود

### ۲. سیستم تگ (کاملاً جدید)
- ادمین از `/admin/tags` می‌تونه تگ بسازه، با نام، رنگ، و توضیحات
  (مثل نمونه‌های پرامپت: Pro Player، Veteran، Helper، Legendary، Holy،
  Troll — این ۶ تا به‌صورت seed داده اولیه ساخته شدن)
- ادمین از صفحه‌ی `/admin/users/<id>` می‌تونه تگ رو به هر کاربری
  اختصاص بده یا ازش بگیره
- تگ‌ها کنار نام کاربر در صفحه‌ی پروفایل نمایش داده میشن

### ۳. بج‌های دستی (مجزا از بج‌های خودکار قبلی)
- بج‌های خودکار (از مرحله ۵) دست‌نخورده باقی موندن — همچنان بر اساس
  فعالیت کاربر خودکار اعطا میشن
- یه نوع بج جدید («دستی») اضافه شد که **فقط** ادمین می‌تونه از
  `/admin/badges` بسازه و از `/admin/users/<id>` به کاربر بده/ازش بگیره
- این دو نوع بج کاملاً مجزا و غیرقابل تداخل هستن (یه چک تو دیتابیس
  تضمین می‌کنه بج خودکار رو نشه دستی اعطا کرد و برعکس)

### ۴. لاگ مدیریتی (Admin Action Log)
- هر اکشن ادمینی روی بج/تگ (اعطا یا برداشتن) در جدول `admin_logs` ثبت
  میشه — چه کسی، چه کاری، روی کی، کِی
- این پایه‌ایه برای گسترش بیشتر لاگ مدیریتی در مراحل بعدی (مثل تایید
  آگهی، تغییر کمیسیون و غیره)

### ۵. Rate Limiting سطح دیتابیس
- محدودیت‌های پایه (دیتابیسی، نه سطح شبکه) روی پست (۵ در دقیقه)، پاسخ
  (۱۰ در دقیقه)، گزارش (۱۰ در ۵ دقیقه)، و پیام چت اتاق (۸ در ۱۰ ثانیه)
- این یه لایه‌ی دفاعی پایه‌ست، نه جایگزین یک سیستم Rate Limiting واقعی
  در سطح Edge/Middleware — ولی جلوی ساده‌ترین حملات spam رو می‌گیره
  حتی اگه آن لایه نبود یا اشتباه تنظیم شده بود

## چی رو تست کنی

### تست رفع باگ‌های امنیتی (مهم‌ترین تست)
1. با یک اکانت عادی (غیر ادمین) وارد شو.
2. باز کن کنسول مرورگر یا یه ابزار مثل Postman، و سعی کن مستقیم با
   Supabase client این رو بزنی: `supabase.from('profiles').update({
   is_admin: true }).eq('id', خودت)`. باید با موفقیت "اجرا" بشه ولی
   `is_admin` همچنان `false` بمونه (چون تریگر مقدار رو برمی‌گردونه).
3. سعی کن `trust_score` خودت رو دستی بالا ببری — همون نتیجه، تغییری
   اعمال نمیشه.
4. سعی کن یه پست رو بدون ادمین بودن پین کنی (`is_pinned: true`) —
   تغییری اعمال نمیشه.

### تست آواتار
1. برو `/settings/profile`، یه عکس آپلود کن.
2. صفحه رو رفرش کن — عکس باید مونده باشه.
3. عکس رو حذف کن (دکمه‌ی X روی پیش‌نمایش) و ذخیره کن — باید به آواتار
   پیش‌فرض (حروف اول اسم) برگرده.
4. یه فایل غیرعکس (مثلاً یه فایل متنی با پسوند جعلی .png) آپلود کن —
   باید رد بشه (هم چک کلاینت هم چک سرور).

### تست تگ و بج دستی
1. با اکانت ادمین، برو `/admin/tags` و یه تگ جدید بساز (یا از تگ‌های
   seed شده استفاده کن).
2. برو `/admin/badges`، یه بج دستی جدید بساز.
3. برو `/admin/users`، یه کاربر رو پیدا کن و وارد صفحه‌ی جزییاتش شو.
4. روی یه تگ و یه بج کلیک کن تا بهش اختصاص داده بشه.
5. برو پروفایل عمومی اون کاربر — تگ باید کنار اسمش، و بج باید تو بخش
   بج‌ها نمایش داده بشه.
6. دوباره تو پنل ادمین، روی همون تگ/بج کلیک کن تا حذفش کنی — باید از
   پروفایل کاربر هم حذف بشه.

## مرحله بعدی

کیف پول داخلی — کاربر شارژ درخواست می‌ده (با مدرک پرداخت)، ادمین تایید
می‌کنه، موجودی اضافه میشه. برداشت هم به همین شکل (درخواست کاربر، تایید
ادمین). این پایه‌ی Marketplace می‌شه که در مراحل بعد از اون میاد.
